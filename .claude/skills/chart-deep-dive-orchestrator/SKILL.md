# Chart Deep Dive Orchestrator

## Phase 0: 컨텍스트 확인

작업 시작 전 `_workspace/` 디렉토리 존재 여부로 실행 모드를 정한다.

- `_workspace/` 없음 → **초기 실행** (Phase 1부터 전체)
- `_workspace/` 있음 + 사용자가 특정 부분만 수정 요청 (예: "지표만 추가해줘", "프론트만 손봐줘") → **부분 재실행** (해당 Phase만 재수행, 아래 "부분 재실행 매핑" 참조)
- `_workspace/` 있음 + 사용자가 새 요구사항/입력 제공 → **새 실행** (기존 `_workspace/`를 `_workspace_prev/`로 이동 후 Phase 1부터)

**부분 재실행 매핑:**
| 요청 내용 | 재호출 대상 |
|---|---|
| 데이터 소스/검색 API 수정 | `stock-data-agent` → 이후 QA → 필요시 `technical-analysis-agent` |
| 지표/패턴/고급기법 추가·수정 | `technical-analysis-agent` → QA |
| 레이아웃/화면 배치 수정 (구현 전) | `frontend-chart-agent`(와이어프레임만) → 사용자 재확인 |
| UI/차트 화면 수정 (구현 후) | `frontend-chart-agent`(구현) → QA |
| "전체 다시 확인해줘" | QA만 전체 재실행 |

## 실행 모드

**서브 에이전트 패턴.** 이 환경에는 팀 통신(TeamCreate)이 없으므로 오케스트레이터가 `Agent` 도구로 각 에이전트를 순차/병렬 호출하고, 산출물은 `_workspace/`에 파일로 남겨 다음 에이전트의 입력으로 사용한다. 에이전트 간 조율이 필요하면(예: QA가 발견한 문제를 원인 에이전트에 전달) 오케스트레이터가 중계한다 — 이전에 스폰한 에이전트를 SendMessage로 재개(resume)해 수정 지시를 전달할 수 있다.

모든 `Agent` 호출에 `model: "opus"`를 명시한다.

## Phase 1: 파이프라인 실행

```
[오케스트레이터]
  1. Agent(stock-data-agent) — 검색 API + KR/US OHLCV 정규화 스키마 확정
     → _workspace/01_stock-data-agent_api-spec.md
  2. Agent(qa-integration-agent) — 스키마 자체 검증(필드 일관성, KR/US 동일 스키마인지)
  3. Agent(technical-analysis-agent) — 1의 스키마를 입력으로 지표/패턴/고급기법 계산
     → _workspace/02_technical-analysis-agent_output-schema.md
  4. Agent(qa-integration-agent) — 경계면 1↔3 검증
  5. Agent(frontend-chart-agent, 와이어프레임 모드) — 1,3의 스키마를 기반으로 화면 레이아웃만 설계 (코드 작성 안 함)
     → _workspace/03_frontend-chart-agent_wireframe.md
  6. [사용자 확인 게이트] 와이어프레임을 사용자에게 보여주고 승인/수정 요청을 받는다 — 승인 없이 다음 단계로 넘어가지 않는다
  7. Agent(frontend-chart-agent, 구현 모드) — 승인된 와이어프레임 + 1,3의 스키마를 기반으로 실제 검색+차트 UI 구현
     → _workspace/04_frontend-chart-agent_notes.md
  8. Agent(qa-integration-agent) — 경계면 3↔7 검증 + 전체 통합 검증
     → _workspace/05_qa-integration-agent_report.md
  9. 결과 종합 및 사용자 보고
```

2, 4는 가벼운 자체/경계면 검증이므로 8의 전체 검증과 중복되면 생략 가능 — 단, "점진적 QA" 원칙에 따라 앞 단계 문제를 늦게 발견하는 것보다 즉시 발견하는 편이 수정 비용이 낮으므로 기본은 포함이다.

**5→6→7이 별도 단계로 나뉜 이유**: 이 서비스는 지표·패턴·갭·하모닉·일목균형표·엘리엇파동·변곡점예측까지 겹쳐 보여줘야 해서 화면이 산만해지기 쉽다. 화면 배치를 코드로 먼저 굳혀버리면 나중에 레이아웃을 바꿀 때 재작업 비용이 크므로, 와이어프레임(텍스트 스케치/ASCII 목업 수준으로 충분하다 — 픽셀 단위 정밀 목업 아님) 단계에서 사용자 확인을 받은 뒤에만 실제 구현으로 넘어간다. 이 게이트는 생략하지 않는다.

## 데이터 전달 프로토콜

- **파일 기반**: `_workspace/{순번}_{agent}_{artifact}.md`에 API 스펙/스키마 문서를 남긴다. 이후 에이전트는 반드시 이 파일을 먼저 읽고 시작한다.
- **반환값 기반**: 각 `Agent` 호출의 반환 메시지로 완료 여부와 산출물 경로를 확인한다.
- 최종 산출물(실제 코드)은 프로젝트 루트에 직접 작성하고, `_workspace/`는 중간 산출물 및 감사 추적용으로 보존한다 (삭제하지 않는다).

## 에러 핸들링

- 에이전트 호출이 실패하거나 QA가 불일치를 발견하면, 오케스트레이터가 원인 에이전트를 SendMessage로 재개해 구체적 수정 지시(QA 보고서의 해당 항목)를 전달한다.
- 1회 재시도 후에도 해결되지 않으면 해당 이슈를 미해결로 최종 보고서에 명시하고 사용자에게 알린다 — 다른 정상 부분의 진행을 막지 않는다.
- 상충하는 데이터(예: 두 에이전트가 스키마를 다르게 이해)는 임의로 하나를 삭제하지 않고 두 버전을 병기해 사용자에게 확인받는다.

## 실행 후 피드백

전체 파이프라인 완료 후 사용자에게 개선점이 있는지 묻는다. 피드백은 아래로 반영한다:
- 결과물 품질(지표 계산 오류, UI 가독성 등) → 해당 에이전트의 스킬 수정
- 에이전트 역할 재조정(예: "데이터랑 분석을 합쳐도 될 듯") → 에이전트 정의 `.md` 수정
- 워크플로우 순서 변경 → 이 오케스트레이터 스킬 수정
- 트리거 누락 → 이 스킬의 description 확장

모든 변경은 프로젝트 `CLAUDE.md`의 변경 이력 테이블에 기록한다.

## 테스트 시나리오

**정상 흐름**: "삼성전자 검색해서 차트 기술적으로 분석해줘" → Phase 1의 1→9가 순서대로 실행되고(6에서 와이어프레임 승인 획득), 최종적으로 검색 가능하고 지표/패턴/고급기법이 표시되는 웹 서비스가 만들어진다.

**에러 흐름**: `technical-analysis-agent`가 `stock-data-agent`의 OHLCV 필드명을 잘못 가정해 QA(4)에서 불일치가 발견됨 → 오케스트레이터가 `technical-analysis-agent`를 SendMessage로 재개해 올바른 필드명으로 수정 지시 → 재검증 → 정상 진행.
