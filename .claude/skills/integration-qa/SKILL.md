# Integration QA

## 핵심 원칙: 존재 확인이 아니라 경계면 교차 비교

파일이 생성됐는지, 함수가 정의됐는지 확인하는 것은 QA가 아니다. 실제로 두 쪽(생산자/소비자)의 코드를 동시에 열어 필드명, 타입, null 처리 방식이 일치하는지 대조한다.

## 검증할 경계면

1. **`stock-data-agent` 출력 ↔ `technical-analysis-agent` 입력**
   - OHLCV 필드명이 실제로 일치하는가 (`close`인가 `closePrice`인가)
   - 날짜 형식이 두 쪽에서 동일하게 파싱되는가
   - 한국/미국 시장 응답이 정말 같은 필드셋을 갖는가 — 하나라도 시장별로 다르면 분석 에이전트가 조건 분기 없이 깨진다

2. **`technical-analysis-agent` 출력 ↔ `frontend-chart-agent` 소비 코드**
   - 프론트엔드가 파싱하는 필드명이 백엔드가 실제로 반환하는 필드명과 정확히 일치하는가
   - 지표가 `null`일 때 프론트엔드 코드가 크래시 없이 처리하는가 (옵셔널 체이닝 유무를 실제로 확인)
   - 패턴 배열이 비어있을 때(`patterns: []`) UI가 정상 렌더링되는가

3. **API 계약 문서 ↔ 실제 구현**
   - `_workspace/*.md`에 문서화된 스키마와 실제 코드가 구현 도중 어긋나지 않았는가 (문서만 보고 판단하지 말고 코드를 직접 실행/호출해 확인)

## 점진적 검증

세 에이전트 중 하나가 산출물을 완성할 때마다 즉시 해당 경계면부터 검증한다. 전체 완성 후 한 번에 몰아서 하면 문제 발견이 늦어지고 수정 비용이 커진다.

## 불일치 보고 형식

발견한 불일치는 반드시 아래 형식으로 구체적으로 기록한다 (막연한 "잘 안 맞는 것 같다"는 보고 아님):

```
- [경계면] frontend가 `macd.signal`을 읽는데 backend는 `macd.signalLine`을 반환함
  → 원인: technical-analysis-agent의 출력 스키마 문서와 실제 구현이 어긋남
  → 재현: GET /api/analysis?symbol=AAPL 호출 후 macd 필드 확인
```

## 흔한 경계면 버그 패턴

- 통화/단위 불일치 (한국 원화 vs 미국 달러를 동일 포맷터로 표시)
- 날짜 타임존 차이로 캔들이 하루 밀려 보임
- 지표 `null`을 프론트엔드가 `0`으로 잘못 렌더링 (RSI 0은 실제 값일 수도 있어 혼동 위험)
- 패턴 탐지 `confidence`가 0~1인지 0~100인지 두 쪽이 다르게 가정
