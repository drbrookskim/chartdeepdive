# Stock Data Fetching

## 왜 정규화가 먼저인가

한국과 미국 데이터 소스는 필드명, 통화 단위, 거래량 단위, 휴장일 캘린더가 전부 다르다. 정규화 스키마를 먼저 고정하지 않으면 분석 엔진과 프론트엔드가 시장별로 분기 처리를 하게 되고, 그 분기가 코드 전체에 퍼져 버그의 원천이 된다. 따라서 데이터를 가져오는 즉시 아래 공통 스키마로 변환한다.

## 공통 OHLCV 스키마

```json
{
  "symbol": "AAPL",
  "market": "US",
  "currency": "USD",
  "candles": [
    { "date": "2026-07-10", "open": 210.5, "high": 213.2, "low": 209.8, "close": 212.1, "volume": 54000000 }
  ]
}
```

- `date`는 항상 ISO 8601 (`YYYY-MM-DD`), 시장 현지 거래일 기준.
- 가격 필드는 항상 숫자(문자열 아님), 소수점 반올림 없이 원본 정밀도 유지.
- `market`은 `"KR"` 또는 `"US"`만 허용.

## 미국 주식 데이터 소스

**중요: `UsStockInfo` MCP는 실제 백엔드 코드에서 호출할 수 없다.** 이 MCP는 에이전트(빌드 시점의 Claude Code 세션)에만 연결된 도구이지, 배포되는 Next.js/Node 서버가 런타임에 호출할 수 있는 API가 아니다. 이 MCP는 오직 **개발 중 참고용**(응답 필드 구조를 미리 확인해 정규화 매핑을 설계할 때)으로만 쓰고, 실제 프로덕션 코드는 서버가 런타임에 직접 호출 가능한 방법으로 구현한다:
- `yahoo-finance2` 같은 npm 패키지 (Yahoo Finance 비공식 API를 감싼 라이브러리, `UsStockInfo` MCP와 유사한 데이터 소스를 직접 호출)
- 또는 Yahoo Finance/Alpha Vantage 등 공개 REST 엔드포인트 직접 호출

반환 필드를 위 공통 스키마로 매핑할 때 필드명이 다를 수 있으므로 실제 응답을 한 번 찍어보고 매핑 테이블을 코드 주석이 아닌 `_workspace` 문서에 남긴다.

## 한국 주식 데이터 소스

현재 프로젝트에 연결된 MCP 중 한국 주식의 실시간/과거 시세를 직접 제공하는 것은 없다. `opendart`는 재무제표·공시 데이터이고 `AptInfo`는 부동산, `NaverSearch`는 뉴스/블로그 검색이지 시세가 아니다.

구현 시점에 아래 중 하나를 조사해 선택한다 (조사 결과와 선택 근거는 `_workspace/01_stock-data-agent_api-spec.md`에 기록):
- 네이버 금융의 공개 시세 엔드포인트 (비공식, 약관 확인 필요)
- 한국거래소(KRX) 정보데이터시스템 공공 API
- `pykrx` 같은 오픈소스 라이브러리 (내부적으로 KRX 데이터를 크롤링)

어느 것을 쓰든 위 공통 스키마로 변환하는 어댑터 함수를 반드시 거치게 한다.

## 종목 검색

검색 API는 티커/종목코드뿐 아니라 종목명(한글/영문) 부분 일치도 지원해야 한다. 검색 결과에는 최소 `{symbol, name, market}`을 포함해 사용자가 시장을 구분해 고를 수 있게 한다.

## 캐싱

- 과거 OHLCV(오늘 이전 데이터)는 변하지 않으므로 장기 캐시(예: 1일 TTL) 가능.
- 오늘 데이터/현재가는 실시간성이 필요하므로 짧은 TTL(수 분 단위) 또는 캐시 없이 매번 조회.
- 캐싱 계층을 데이터 소스 어댑터와 분리해, 소스가 바뀌어도 캐싱 로직은 그대로 재사용되게 한다.
